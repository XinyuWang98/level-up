**题目一：复杂 SQL – 用户连续活跃与转化归因**
**场景**：现有三张表：1）用户登录流水表 `login_log`（user_id, login_time）；2）广告曝光表 `ad_view`（user_id, ad_id, view_time）；3）订单表 `orders`（user_id, order_time, order_amount）。广告系统在 2023-06-01 至 2023-06-30 期间上线了新策略，希望分析该策略对用户长期活跃和转化的影响。
**问题**：
1. 计算在 2023-06-01 当日首次登录的用户，在后续 30 天内每月的最大连续活跃天数（连续活跃定义为至少有一次登录）。
2. 对于在 6 月份有广告曝光的用户，将最后一次曝光广告 ID 视为最终归因点。请计算每个广告 ID 带来的首次下单用户数（需去重），且要求下单时间在曝光时间之后的 7 天内。
3. （附加）若广告曝光可能存在同一用户多次曝光不同广告，请用窗口函数实现最终点击归因（即取最接近下单的一次曝光，且时间差不超过 7 天）。
**要求**：使用 Spark SQL 语法，考虑性能与数据倾斜，写出完整查询。

**破题思路与考察点**：
- 连续活跃天数：需用窗口函数按 user_id 分组，对 login_time 排序后，用日期差与行号差做连续性判断，再用分组聚合取最大连续段。
- 归因逻辑：需先获取每个用户的最后一次曝光（或最近一次曝光），再关联订单，并做时间窗口过滤。
- 性能：需注意用户去重、分区裁剪（按时间）、避免笛卡尔积。若数据量大，可先聚合再关联。
**参考答案红线**：
- 未使用窗口函数（如 ROW_NUMBER）处理归因优先级 → 逻辑错误。
- 计算连续活跃时未处理跨月问题（如使用 DATEDIFF 和 row_number 差值） → 结果不准确。
- 未考虑数据倾斜（如大 user_id 的登录记录过多），未使用 mapjoin 或分桶优化 → 工程思维不足。

**题目二：大数据/Python 工程 – 海量交易数据内存优化与倾斜处理**
**场景**：给定一个约 500GB 的交易流水文本文件（格式：order_id, user_id, amount, city_id, dt），需在有限内存（如 8GB）的集群节点上完成以下操作：
1. 统计每个城市（city_id）的订单金额 Top 10% 的高价值订单（即按城市分组，取 amount 的 90 分位数以上）。
2. 计算每个用户的累计交易金额，并按金额分布将用户分为 5 个等级（等频分桶）。
3. 发现 user_id 分布极不均匀（部分大卖家占比超过 50% 交易量），请设计优化方案避免 Spark 计算倾斜。
**要求**：写出 PySpark 代码，并说明每个优化步骤的原理。

**破题思路与考察点**：
- 内存优化：采用 DataFrame API 而非 RDD，使用 parquet 格式存储中间结果，避免 collect() 操作。
- 分位数计算：使用 approxQuantile 或分桶后计算，避免全量排序。
- 数据倾斜处理：识别倾斜 key（如 sample 采样），采用加盐（salt）打散再聚合、两阶段聚合（局部聚合+全局聚合）、或倾斜 key 单独处理。
**参考答案红线**：
- 直接使用 pandas 读取全量数据 → 内存溢出。
- 未处理倾斜，直接 groupBy 导致少数 task 卡住 → 缺乏生产经验。
- 分位数计算采用全量排序（如 orderBy + rank） → 性能差。

**题目三：A/B 实验底层统计学 – 方差缩减与 SRM 排查**
**场景**：某电商大促期间，你在实验平台发现一个核心指标的 A/B 实验组和对照组在第一天就出现显著差异（p-value < 0.001），但后续几天差异消失。实验分配为均匀哈希分流，日志上报完备。
**问题**：
1. 请列举可能导致这种“显著-消失”现象的三种原因，并设计排查步骤。
2. 若实验指标为转化率（二分类变量），原计划采用卡方检验，现希望使用 CUPED 进行方差缩减。请说明 CUPED 在此场景下的具体实施步骤，包括协变量选择依据、数学公式，并解释为何能提升灵敏度。
3. 若实验平台监控到 SRM（Sample Ratio Mismatch）报警，请描述从数据上报到实验配置的全链路排查框架。

**破题思路与考察点**：
- 原因推测：可能包括初期用户兴奋效应、实验组对照组流量不均匀、埋点上报延迟或丢失、外部活动干扰等。
- CUPED 实现：需选择与目标指标相关且不受实验影响的协变量（如实验前的转化率），通过线性模型调整后得到方差缩减的指标，公式：Y_cuped = Y - θ*(X - E[X])。
- SRM 排查：从哈希种子、分流逻辑、用户标识一致性（user_id/device_id）、数据上报丢失/重复、实验配置错误（流量重叠）等方面构建排查树。
**参考答案红线**：
- 未考虑外部混淆因素（如大促时间效应） → 业务敏感度不足。
- 未说明 CUPED 中协变量需满足“与实验分配独立” → 统计基础不牢。
- SRM 排查仅停留在“查看日志”层面，未形成系统化链路 → 缺乏平台建设经验。

**题目四：因果推断/准实验设计 – 无法 A/B 时的策略评估**
**场景**：公司针对“高价值用户”（历史消费 top 10%）推出了专属 VIP 客服通道（非随机，由运营根据规则手动授予）。现希望评估 VIP 客服通道对用户留存（90 天复购率）的真实影响。由于资源有限，无法对所有高价值用户随机开通。
**问题**：
1. 设计一个基于观测数据的评估方案，要求说明方法选择（如 DID、PSM、RD）及理由。
2. 若选用 PSM，请详细描述如何构建倾向得分模型（包括特征选择、模型训练、平衡性检验），以及匹配后的效果估计方法。
3. 指出该评估中可能存在的因果推断陷阱（如未观测混淆变量）及缓解思路。

**破题思路与考察点**：
- 方法选择：由于处理组和对照组非随机分配，但存在时间维度（开通前后），DID 是常见选择；若开通规则明确（如严格按消费阈值），可使用断点回归（RD）；若规则复杂，可用 PSM 匹配相似用户。
- PSM 实施细节：特征应包括历史消费、活跃度、用户属性等；使用逻辑回归或树模型预测倾向得分；匹配后需检查标准化差异（SMD）<0.1；效果估计可用 ATT（平均处理效应）。
- 陷阱与缓解：可能存在选择偏差（如更忠诚用户更可能被选中），可尝试使用工具变量（IV）或敏感性分析（如 Rosenbaum bounds）检验结论稳健性。
**参考答案红线**：
- 直接比较处理组和对照组均值 → 忽略选择偏差。
- 未进行匹配后的平衡性检验 → 方法执行不完整。
- 未讨论未观测混淆因素（如用户隐性需求）的威胁 → 因果推断深度不够。

**题目五：开放式宏观商业决策 – 指标异动归因与北极星指标设计**
**场景**：你是某内容平台的数据负责人，发现平台“日均用户使用时长”在最近 4 周连续下跌 15%，但同期 DAU 基本稳定。平台近期上线了以下功能：1）信息流算法升级；2）新增“短视频”频道；3）加强弹窗通知促活。
**问题**：
1. 请设计一个系统性的异动归因分析框架，包括数据拆解维度、假设清单、验证方法。
2. 若发现使用时长下降主要来自老用户，且集中在晚间时段，请提出 2-3 个深度假设，并说明如何通过数据验证。
3. 现有北极星指标“日均使用时长”是否合适？如果不合适，请提出一个新的北极星指标，并说明设计理由（需考虑平台长期生态健康与商业变现）。

**破题思路与考察点**：
- 归因框架：需从用户分群（新/老、高低活）、内容类型（图文/视频）、时段、功能曝光等维度下钻；构建假设树（如算法改动导致兴趣匹配度下降、短视频侵蚀图文时长、弹窗引起反感）；通过 AA 实验、用户调研、因果图验证。
- 深度假设：如“算法升级导致信息茧房，老用户审美疲劳”、“短视频频道占用入口但消费深度不足”、“晚间推送过于频繁导致用户主动避开”。
- 北极星指标设计：需平衡用户价值（如留存、满意度）、平台生态（如内容多样性、创作者激励）、商业目标（如广告收入）；可改为“人均优质内容消费时长”或“7 天留存率”，并配套护栏指标（如创作者数量、用户反馈负面率）。
**参考答案红线**：
- 归因仅停留在整体指标，未拆解到用户群与场景 → 分析粗糙。
- 北极星指标仅考虑单一目标（如最大化时长），忽略生态健康 → 战略视野不足。
- 未提出可落地的验证方案（如 AB 实验、用户访谈） → 纸上谈兵。
