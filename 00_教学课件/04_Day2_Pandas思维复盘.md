## 1. 核心世界观差异

| 维度         | Week 1: Python 原生                                  | Week 2: Pandas                                           | SQL 类比                 |
| :----------- | :--------------------------------------------------- | :------------------------------------------------------- | :----------------------- |

## 2. 今天的三个“冲击点”解析

### 💥 冲击 1：为什么不用 `for` 循环了？
*   **Week 1**: 想让每个价格打八折？
    ```python
    res = []
    for p in prices:
        res.append(p * 0.8)  # 必须一个个算
    ```
*   **Pandas**:
    ```python
    df['price'] * 0.8  # 一行代码，Pandas 自动把乘法“广播”给这列的 100 万行
    ```
*   **结论**：在 Pandas 里写 `for` 循环是“违法行为”（效率极低）。

### 💥 冲击 2：为什么新增列像写字典？
*   **你的直觉**：`df.add_column(...)` 或者 `df.append(...)`
*   **实际做法**：`df['new_col'] = ...`
*   **底层逻辑**：Pandas 的 DataFrame 本质上是一个 **"Key 是列名，Value 是整列数据"** 的超级字典。所以这一招沿用的是 Python 字典的 `d[key] = val` 逻辑。

### 💥 冲击 3：函数的“反人类”括号
*   **困惑**：为什么是 `groupby('sex')['tip'].mean()` 而不是 `mean(tip)`？
*   **原因**：Python 是面向对象语言。
    *   `df` 是一个对象。
    *   `groupby` 是它的方法，返回一个新的“分组对象”。
    *   `mean` 是那个分组对象的方法。
    *   **链式调用 (Chaining)**：像接力棒一样，一个传一个。

## 4. 进阶答疑：关于“往原表加列”的执念

**Q: 我喜欢不断往原表里加新列 (`df['new'] = ...`)，这是 SQL 旧思维吗？**

**A: 绝对不是！这是标准的“宽表分析模式” (Wide Table / Enrichment Pattern)。**

| 模式         | **宽表模式** (你的做法)                                                  | **链式模式** (工程做法)                                    |
| :----------- | :----------------------------------------------------------------------- | :--------------------------------------------------------- |
| **写法**     | `df['A'] = ...`<br>`df['B'] = ...`                                       | `df.assign(A=...).assign(B=...)`                           |
| **特点**     | **步步为营**。每算一步，表就“胖”一圈。你可以随时停下来检查 `df.head()`。 | **一气呵成**。不产生中间变量，一条管道通到底。             |
| **适用场景** | **探索分析 (EDA)**。<br>你想搞清楚每一个指标的分布，需要反复回头看数据。 | **生产代码 (ETL)**。<br>跑完就扔，不需要人工检查中间过程。 |

**结论**：
在 Jupyter Notebook 里做分析，**宽表模式是王道**。
只有当你写的脚本要自动处理 10 亿行数据时，才需要考虑“不加列以节省内存”的链式写法。现在？尽情地加列吧！
