# /generate_exam 工作流

## 1. 触发与筹备
当用户请求生成笔试题时，Agent 必须通过提问或读取上下文，明确：
- **目标公司名称 (Target Company)**
- **具体岗位 JD (Job Description)**

## 2. 外部扫描与考点汇编
在出题之前，Agent **必须首先运用 Web Search 工具或读取项目知识库**，检索由其他候选人分享的面经、笔试总结或该公司该岗位的技术栈偏好（例如：“某短视频大厂B公司 资深数据分析师 笔试 考点”）。

这能确保生成的题目不是泛泛而谈的 LeetCode 原题，而是**高度贴合真实业务环境**的场景化考点。

## 3. 生成试卷逻辑
1. 将提取到的目标公司考向与 JD 中的硬指标（如：熟练使用 Hive SQL、精通随机森林和 A/B 实验）相结合。
2. 构造 **3-5 道高质量综合大题**。
3. **出题风格要求 (Style Guideline)**：
   - 拒绝选择题或死板填空题。
   - 提供复杂的、真实的（经过脱敏处理的）业务背景前提（如：电商平台大促、短视频流式推荐系统冷启动等）。
   - 让考察从基础语法延伸到业务分析推演与代码工程化能力。

## 4. 标准化输出格式

### 📝 [目标岗位] 场景化模拟笔试考卷

**[考试说明与时间分配建议]**
(说明试题结构与难度设定)

---

#### 🔵 题号 [01] - [核心考察点：如 Hive/Spark 亿级日志处理]
**业务上下文 (Business Context)**:
[带入具体的业务场景，给出几张相关联数据库表的 Schema 及字段说明]

**业务问题 (The Problem)**:
- [子问题 1: 复杂窗口函数/聚合计算]
- [子问题 2: 业务指标异动监控或 SQL 性能调优]

#### 🔵 题号 [02] - [核心考察点：如 AB 实验/策略评估]
**业务上下文 (Business Context)**:
[描述一个由于各种现实原因难以直接做随机试验的策略]

**业务问题 (The Problem)**:
- [子问题 1: 在此情境下，选择何种因果推断方法（PSM/DID/Delta Method）并阐述业务逻辑]
- [子问题 2: 编写关键数据处理/建模代码段的实现（提供 Pandas/DoWhy 等对应框架）]

---

**答题指南 (Submission Guide)**:
请用户将作答整理在一个独立文件中提供给 Agent。系统收到后将以资深面试官的苛刻视角进行批注和代码评审 (Code Review)。
